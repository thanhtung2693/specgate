The governed loop in three moves.

SpecGate does not replace your spec tool, tracker, or IDE. It makes the handoff explicit, then checks the finished work against the approved contract.

  1. Publish

    Bring any spec format

    OpenSpec, Spec Kit, a quick change note, or your own document set. A profile declares roles, gates, and required topics.

    spec packagev0.3 # Audit-log permissions
    ## Scope / non-goals
    ## Acceptance criteria
    [ ] Admins can view audit events
  2. Hand off

    Approve one Context Pack

    The coding agent gets the approved version, acceptance criteria, non-goals, risks, tasks, and applicable Skills through the CLI.

    context-pack.mdapproved $ specgate work context WI-101
    contract, criteria, risks, tasks
    for Cursor / Claude Code / Codex
  3. Verify

    Review delivery evidence

    SpecGate compares completion claims, tests, docs, and changed files against every acceptance criterion before you close the work.

    delivery reviewpass 4/4 criteria met
    tests, types, lint verified
    evidence recorded per criterion

    What teams ask first.
    
        
    
        
    
          
    
            How is this different from spec tools?
            
    Those help shape specs in your repo. SpecGate accepts the output, versions it, runs readiness gates, creates the Context Pack, and verifies delivery evidence against the approved contract.
    
          
    
          
    
            What spec formats do you accept?
            
    Any document set your team can name clearly. Bring an OpenSpec or Spec Kit project, Markdown notes, or custom docs. A profile maps documents to roles and decides which gates apply.
    
          
    
          
    
            Do we have to leave our coding agent?
            
    No. Cursor, Claude Code, Codex, or a human can use the CLI. The agent pulls the Context Pack before coding and reports completion evidence when the build is ready for review.
    
          
    
          
    
            Do I need an LLM API key?
            
    No. The deterministic gates need nothing, and the LLM readiness gates and delivery review can run through your IDE coding agent. Configure a server-side model (OpenAI, Anthropic, Google, or OpenRouter) only when you want those steps to run without a coding agent in the loop.
    
          
    
          
    
            Does it replace Linear, GitHub, or GitLab?
            
    No. Issues, sprints, PRs, and assignment stay where they are. SpecGate owns the approved contract, Context Pack, delivery review, and proof. Experimental connectors mirror tracker status one way.
    
          
    
          
    
            What does "verify" actually check?
            
    Each acceptance criterion is checked against reported evidence and automated checks such as tests, types, and lint. The verdict records what is met, unmet, or unclear, then carries failed criteria into the next Context Pack.
    
          
    
          
    
            Which trackers do you connect?
            
    Experimental connectors are available for Linear, GitHub, and GitLab. One integration per provider. Status mirrors from tracker to SpecGate so there is no second source of truth.
    
          
    
          
    
            What's supported in the alpha vs experimental?
            
    The supported alpha core is the governed loop: gates, Context Pack, delivery review, evidence trust, and CLI + IDE handoff. Integrations, knowledge search, governance chat, and the web UI are experimental.
    
          
    
          
    
            How do I know it's working?
            
    Run specgate stats. It uses real gate runs and delivery reviews to show first-pass yield, catches before build, catches before merge, rework depth, ambiguity saves, and cycle time. With no data it says "not enough data yet".
