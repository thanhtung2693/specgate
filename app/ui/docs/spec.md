# SpecGate UI Specification

## Overview

The SpecGate UI is the experimental product shell for governed delivery. It presents work items, review queues, artifacts, settings, and the governance agent in one responsive interface. Context Packs, gates, delivery evidence, and skills are surfaced in the work-item context where they are acted on. The primary authoring and implementation workspace remains IDE + CLI.

## Routes

| Path | Purpose | Required behavior |
| --- | --- | --- |
| `/` | Default entry | Redirects to `/work` |
| `/work` | Governed work overview | Shows queue views, reason filters for attention work, a board/list switch, lifecycle lanes, and an optional list queue |
| `/work/:itemKey` | Work item detail | Shows route-backed work item cockpit with Overview, Handoff, Verification, and Activity tabs; registry-backed routes show a loading state while the workboard is fetching, and unknown keys show an explicit not-found state with a Back to work action |
| `/reviews` | Human review queue | Shows a derived triage queue for delivery reviews, gate failures, evidence gaps, and ready review handoffs |
| `/artifacts` | Versioned governance artifacts | Shows an artifact list with selected artifact metadata, document files, and expected gates |

Unknown section routes redirect to `/work`.

## Layout

- Desktop widths (`1024px` and up) use a persistent vertical shadcn `Sidebar`.
- Narrow laptop, tablet, and mobile widths use the sidebar's built-in sheet behavior so dense Work, Reviews, and Artifacts tables keep the full viewport instead of being squeezed beside navigation.
- The browser document title is `SpecGate (Experimental)`.
- The sidebar product title is `SpecGate (Experimental)`.
- The app mark and favicon use the landing-page logo asset.
- The governance agent opens from a floating assistant-ui modal trigger on desktop, tablet, and mobile. The trigger sits above page content but below modal overlays so it does not cover document inspectors, settings dialogs, or review footers.
- The governance agent header exposes a compact thread-history page for conversation catch-up and thread management.
- Header includes the sidebar toggle, section title, section description, and theme toggle.
- Settings does not embed the governance chat inside configuration panels.
- Workflow navigation contains Work, Reviews, and Artifacts. First run without a local UI session bootstraps one automatically — identity is local attribution only, so no form gates the app. When Doc Registry is configured, the shell reuses the first existing registry workspace (default name "SpecGate" if none) and calls `POST /api/v1/identity/bootstrap` with the default `dev` user, storing the returned selection in browser localStorage; if the registry is unavailable, a browser-local default session ("Dev" in "Local Workspace") is used instead. No bundled sample workspaces exist. The sidebar footer owns local-user actions: change name always, change workspace only when more than one workspace exists. There is no logout — a local attribution session has nothing to sign out of. Settings remains a separate sidebar-footer affordance beside the local user/workspace block. Context Packs, gates, delivery, and skills remain contextual surfaces.

## Theme

The app uses semantic Tailwind tokens in `src/index.css`. Tokens are derived from the landing-page palette and support light and dark modes through `next-themes`.

## Governance Agent

The governance agent uses `@assistant-ui/react` primitives. The shell owns placement, responsive behavior, composer affordance, and transcript rendering. Contextual actions that send prompts to the governance agent use the robot icon so they are visually distinct from navigation, file, and copy actions. Governance-agent actions are advisory: summarizing gates, explaining blockers, drafting checklists, evaluating handoff readiness, and proposing next governance steps. Durable workflow transitions such as creating work, approving or rejecting artifacts, running gates, persisting gate snapshots, creating tracker handoffs, copying Context Pack URIs, and IDE implementation handoff must be direct UI/backend/CLI actions rather than chat-only buttons. Compact or boundary-sensitive actions expose concise tooltips that clarify owner and outcome, especially when an action opens the governance agent, copies handoff material, or persists registry state.

Runtime selection is environment-driven. When `VITE_LANGGRAPH_API_URL` is set, the agent uses `@assistant-ui/react-langgraph` with the `governance` graph and requests `messages` plus `updates` stream modes. In Vite dev, the recommended value is the same-origin proxy path `/api/agents`, backed by `LANGGRAPH_PROXY_TARGET`, so thread-history and stream calls do not depend on browser CORS across localhost ports. When it is unset, the agent uses the deterministic local runtime so the app shell remains usable without the agents service. Runtime names are implementation details and must not appear in the visible chat UI.

The thread-history surface is a separate in-panel page, not a drawer stacked above the chat transcript. The thread page supports the smallest useful conversation workflow: show available governance-agent threads, search visible thread titles, switch to a listed active thread, rename a thread inline, archive a thread, browse archived threads through the same local search, restore an archived thread, delete a thread after confirmation, start a new thread, and return to chat with Back navigation. Thread rows show a readable title and relative activity time, never raw local or remote thread ids. LangGraph runtime history is backed by `client.threads.search()` for threads with `source=specgate-ui` and `surface=governance-agent`, so reloads can recover SpecGate-created governance conversations without Assistant Cloud. Rename and generated titles persist into LangGraph thread metadata; archive persists as `metadata.archived=true`; restore/unarchive persists `metadata.archived=false`; delete calls the LangGraph thread delete API. Archived rows are secondary conversation-history rows: they can be restored or deleted, but they do not expose workflow actions or imply any work-item, gate, artifact, evidence, Context Pack, or delivery state has changed. The governance-agent chat panel does not fetch or render thread-to-artifact links; artifact context comes from work items, Context Packs, artifact search, or explicit artifact ids. The transcript renders only visible user/assistant content; reasoning-only assistant chunks stay represented by the single `Thinking...` state instead of blank message rows. Runtime stream errors are shown in the composer area with a short actionable explanation, including provider rate-limit failures. The LangGraph thread-list adapter calls the governance custom route `POST /governance/threads/{thread_id}/title` when assistant-ui asks for a title, then persists the returned title into LangGraph thread metadata for future loads. Cross-workspace history and URL-addressable thread routes are deferred until product need is clear.

Assistant replies render as plain transcript text rather than boxed chat bubbles. While the run is active, the thread shows a small shimmering `Thinking...` state. The composer exposes Stop through `ComposerPrimitive.Cancel` while the assistant is running.

The composer supports a `/` trigger popover that runs command-style governance prompts for handoff, gates, and evidence.

Plain Enter sends the governance-agent message. Shift+Enter inserts a newline.

Backend runtime adapters must not change the app shell's layout contract.

The production Docker image is built by `docker/Dockerfile.ui`. It serves the Vite bundle through nginx and must provide SPA fallback for route-backed pages. Release Compose runs the same image as `ui`, and the nginx runtime proxies `/api/doc-registry` to Doc Registry and `/api/agents` to the agents service so the compiled browser bundle can use same-origin API paths.

## Data Boundary

`src/data/workboard.ts` is the workboard adapter boundary. It maps Doc Registry `GET /workboard/change-requests` responses into the shell's work-item, lane, signal, and Context Pack summaries.

The Work surface supports queue search across visible work identifiers and metadata, one primary queue-view filter, compact reason filtering for `Needs attention` and `Blocked`, and a board/list switch. Queue chips include a `Delivered` view, positioned before `All work`, for change requests whose server-derived board phase is `delivered` (latest delivery review passed). The delivered phase is additive: old servers never send it and nothing else changes. Delivered items map to a passed delivery with no blocker and are excluded from the `Needs attention` view. The work queue table has no Owner column — assignment is not a product feature and the registry has no owner beyond `created_by` attribution. The list view is the default and shows filtered work in a compact table for dense scanning. The board view is a secondary visualization and shows filtered lifecycle lanes. Filters combine predictably with `AND` semantics. The overview layout keeps the Governance Board surface unframed: clickable workspace statistic cards appear first, then search, queue filters, the reason dropdown, and the view switch appear below them. Search and the Board/List switch have equal control height. The overview does not show a selected-item preview column. Workspace signal metrics are clickable when they map cleanly to an existing queue filter: ready handoff, gate failures, and blocked work. Kanban lane items render as compact cards with gate, delivery, blocker, and updated signals without introducing fields that are not yet present in the workboard contract. In the list table, gate state is displayed beside the work item title rather than as a separate column, and updated time is the final column. When Doc Registry is configured, the initial and error states must not flash bundled sample work; they show an empty/loading live workspace placeholder with a CLI-oriented next step instead. The UI does not create work items; the main authoring workspace is IDE + CLI. Work items enter the board through CLI, IDE plugin, Doc Registry, tracker, or governance-service flows. Work items without a Context Pack expose an advisory Ask what blocks handoff action that opens the governance agent; it must not claim to generate or persist a Context Pack. Before a Context Pack exists, the Work detail Overview tab can check the route classifier through `POST /workboard/change-requests/{id}/classify-route` and persist a human-confirmed quick/full route through `PATCH /workboard/change-requests/{id}`. Route confirmation does not fall back to session-only state; if Doc Registry persistence is unavailable, the UI tells the user that route changes require the registry service. Route controls disappear after handoff because the Context Pack owns the implementation contract. Quick-route work items without a Context Pack expose a direct Create quick Context Pack action in the Handoff tab. That action must confirm with the user, call the agents backend `POST /workboard/change-requests/{id}/context-pack` with `quick_mode=true`, require a returned persisted `change_request`, attach the returned Context Pack to the work item, and make the existing copy/download handoff actions available without implying that implementation or tracker handoff has started. Artifact-only or otherwise incomplete responses show an error and do not create browser-local handoff state.

The Work board also carries a read-only, collapsed-by-default "Governance stats · last 30 days" disclosure below the workspace statistic cards. Expanding it (and only expanding it — the board never fetches stats on load) reads the governance-value projection from Doc Registry `GET /api/v1/stats?days=30`, scoped by the same registry workspace id the workboard uses, through `src/data/stats.ts` (`useGovernanceStats`). The expanded content renders six compact numbers — first-pass yield (success at ≥80%, warning at ≥50%, danger below), pre-build gate catches, post-build review catches with the fixed count, rework runs, ambiguity blocks, and average cycle time — plus a compact catch ledger whose rows show relative time, a work-item key link to `/work/{key}?tab=verification`, a kind badge, and a truncated detail. Zero reviewed items render a single not-enough-data line instead of percentages, and fetch failures render a single stats-unavailable line; the card has no charts, mutations, or synthetic numbers.

When `VITE_DOC_REGISTRY_URL` is configured, Work, Reviews, Artifacts, and work-item detail surfaces read from Doc Registry. In Vite dev, the recommended value is the same-origin proxy path `/api/doc-registry`, backed by `DOC_REGISTRY_PROXY_TARGET`, so the UI can move between local compose ports without CORS failures. The Work adapter prefers the active local UI session workspace id and passes it to `GET /workboard/change-requests?workspace_id=...`, so the browser board matches the user's selected workspace and CLI-scoped intent. Browser-local workspace ids such as `local-*` are not sent to Doc Registry; if the session has no registry workspace id, the adapter discovers the first registry workspace through `/api/v1/workspaces`; if no workspace can be discovered, it falls back to the unscoped registry board. Browser read-after-write overlays for persisted route or Context Pack updates are cleared when the active workspace changes, so a successful mutation from one workspace cannot appear in another workspace's board while the registry reloads. Work-item detail also hydrates normalized acceptance criteria, next actions, gate runs, tracker links, linked WorkBoard Feature context, CR-scoped freshness warnings from `GET /workboard/stale-warnings?change_request_id=...`, evidence manifests from `GET /api/v1/work-items/{id}/evidence`, deterministic evidence gate summaries from `GET /api/v1/work-items/{id}/evidence/gates`, the object-scoped governance explanation from `GET /api/v1/work-items/{id}/policy`, and latest persisted delivery-review detail from `GET /api/v1/work-items/{id}/delivery-status?detail=true`. Quick-route work items and artifacts may be featureless; the browser must not invent Feature ids, names, summaries, or links for them. Live linked Feature context requires a registry `id` or `key` before rendering; incomplete feature responses are ignored instead of becoming a generic `feature` row. Live gate-run rows require a real registry `id` before rendering; incomplete gate-run rows are ignored instead of becoming fake persisted gate history. Live tracker-link rows require both a registry identifier and URL before rendering; incomplete tracker rows are ignored instead of becoming fake `tracker-*` links or `#` actions. Live evidence manifests require real registry `record_id` and `manifest_id` values before rendering; incomplete manifests are ignored instead of becoming fake evidence rows. Live governance policy lineage requires real policy keys before rendering; incomplete lineage rows are ignored instead of becoming fake `policy-*` provenance. Freshness signals are display-only in the Overview tab for stale handoffs, tracker contradictions, and external delivery signals; the browser must not regenerate Context Packs, rerun gates, resolve tracker conflicts, promote artifacts, or submit evidence from a stale-warning row. Evidence, policy, and delivery-review readback are display-only in the Verification tab: the browser must not create evidence manifests, manage submitter/admin credentials, accept exceptions, switch policy profiles, trigger delivery review, rerun gates, or imply that a review has been requested. The Feature model, canonical artifact pointer, and Feature Summary API remain supported backend contracts. The UI does not restore the old top-level Features page, but the Artifacts library includes a read-only Features view for lookup and feature-detail inspection. Feature detail owns the Feature summary, canonical artifact pointer, summary source version, and related artifact list; artifact detail links into Feature detail instead of embedding the long summary when the artifact is feature-backed. External-LLM summary drafting stays a governance-agent/backend capability with deterministic fallback; the UI must not expose a regenerate action unless it calls that backend route and makes review ownership clear. Bundled sample data must not populate Work, Reviews, Artifacts, Features, or work-item detail surfaces; loading, empty, and error states show setup-oriented or live-mode placeholders instead of sample rows, sample acceptance criteria, or sample document bodies. Backend adapters must preserve the route and layout contracts when extending this boundary.

Work-item acceptance, gate next-action, gate-run, tracker-link, linked-feature, policy, evidence, and delivery-review readback failures show explicit live-mode errors instead of hiding their sections, implying that no records exist, or falling back to workboard/sample summaries.

The Verification tab is verdict-first: the Delivery review section renders before Gate state, which renders before the policy explanation, because Reviews deep-links into this tab to read the delivery outcome. Gate state renders collapsed by default as a one-line summary (for example `6 gates · 1 passed · 5 not applicable`) with the aggregate tone; a disclosure expands the full per-gate cards, and `not_applicable` cards render muted. The Gate state header sub-line reads "What each gate checked and why, per run." Expanded gate cards and gate-run rows show the shared gate-catalog sentence (`gateCatalog` in `src/components/layout/shared.ts`) — one plain sentence per gate covering what it verifies and what it reads. Gate run history shows only the latest run per gate by default; a `Show all runs` disclosure reveals the full chronological list. Each persisted gate run with parseable `evidence_json` gets a "Why" disclosure that shows who evaluated it ("Evaluated by platform model (<judge_model>)" for platform LLM judges, "Agent-attested" for `judge_model=agent_attested` / `executor=ide_agent` results), the recorded confidence when it is a judgment, and the evidence content: plain quotes render as quoted text, per-topic completeness detail and delivery-review criteria/checks render as compact label/state/why rows. Deterministic runs whose envelope carries no judgment or evidence content get no disclosure; unparseable evidence renders nothing rather than raw JSON.

Overview acceptance criteria derive their check state from the latest delivery verdict when the delivery-status readback carries per-criterion results: a criterion is matched by criterion id first, then exact text, is checked when its verdict is `met`, and shows a small met/unmet/unclear badge. The registry `done` flag remains the fallback only when no verdict covers the criterion. The section header count (and the Handoff acceptance count) follow the same derivation. The Work context panel shows route and blocker only — no Owner field and no detail-source field.

Delivered work items switch the detail header's primary action to `View verdict`, which opens the Verification tab, and swap the right-rail advisory prompt to a delivery-review summary prompt instead of handoff-blocker questions.

Registry timestamps render through `src/lib/format.ts` with one compact absolute date-time format across Work, Reviews, Artifacts, and gate history surfaces. Dense queue cells may use relative labels when horizontal space is tight, but the absolute timestamp remains available in the detail surface or cell title. Invalid labels fall back to readable text instead of leaking raw placeholders.

Work-item Handoff tabs expose the Context Pack URI, source/evidence summary, route and acceptance status, and local copy/download actions. Copy URI writes the CLI-canonical `specgate://context-pack/...` URI, using the registry change-request id when present and the display key only for locally scoped, unsynced work references. Copy handoff and Download `.md` produce the same markdown summary for CLI handoff and include the executable `specgate work context <ref>` command. IDE coding agents consume this handoff contract outside the UI: they read the Context Pack, edit the repository, run verification, update docs, and report delivery evidence through CLI/MCP flows. The Work detail right rail stays below the main content until very wide desktop viewports so MacBook-width screens do not squeeze or horizontally overflow the page.

For live registry work items with a Context Pack, the Handoff tab reads
`GET /api/v1/work-items/{id}/context-pack` and shows the assembled Markdown as
the canonical preview. Copy handoff and Download `.md` use that registry
Markdown when it is available; the browser-built summary is only a fallback for
non-registry/local preview mode. If live canonical Context Pack readback fails,
Copy handoff and Download `.md` stay disabled while Copy URI remains available.
Unassembled or empty-markdown canonical responses count as unavailable.
Context Pack warnings and knowledge provenance are display-only. The browser
must not regenerate Context Packs, rerun gates, or
submit delivery evidence from the preview.

`src/data/artifacts.ts` is the artifact adapter boundary. When `VITE_DOC_REGISTRY_URL` is configured, the Artifacts surface reads `GET /artifacts?limit=50`, same-feature versions from `GET /artifacts?feature_id=...`, the selected artifact's `GET /artifacts/{id}/files` document list, document content from `GET /artifacts/{id}/files/_?path=...`, feature reference attachments from `GET /features/{feature_id}/attachments`, artifact-linked feedback from `GET /governance/feedback-events?artifact_id=...`, persisted readiness history from `GET /artifacts/{id}/readiness-runs?limit=20`, saved draft revisions from `GET /artifacts/{id}/revisions`, artifact-scoped audit events from `GET /events?artifact_id=...&limit=20`, the persisted artifact policy explanation from `GET /api/v1/artifacts/{id}/policy`, and the read-only expected gate preview from `GET /api/v1/artifacts/{id}/gate-preview`. When the base URL is unset, the page shows setup-oriented empty placeholders instead of representative sample artifacts. Request failures and empty responses show live-mode empty/error placeholders instead of representative sample artifacts. Live artifact rows require a real registry artifact id, live document rows require a real registry file path, linked Feature rows require a real registry `id` or `key`, and live artifact evidence rows require real registry row ids for attachments, feedback, readiness runs, audit events, and saved revisions before the browser renders them or issues follow-up artifact/file requests; expected gate preview rows require a real `gate_key`. The UI must not synthesize artifact ids, document paths, feature ids, gate keys, or evidence row ids from incomplete registry rows. Artifacts are document bundles: the library supports forgiving search across visible artifact identifiers plus status chips, the selected artifact is restorable through `?artifact=<artifact_id>`, and row selection opens a compact detail modal with metadata, documents, feedback, read-only attachments, readiness history, saved revisions, audit events, expected gates, governance policy snapshot, a conditional Open work action, and an advisory Ask agent action. Featureless quick artifacts are labeled as standalone quick-path artifacts and do not show Feature context, Feature summary, related-feature grouping, or a fake Feature identifier. The Features view uses the same search control for registry Feature lookup; selecting a feature opens a compact read-only detail modal with status, version, summary source, canonical artifact, summary Markdown, related artifacts from the currently loaded artifact library with the canonical artifact first, Open work when the feature key looks routeable, and Open artifact actions for related rows. Open work is shown only when the artifact or feature identifier is a known work-item key shape; otherwise the modal avoids linking to a dead or ambiguous work route. Documents render as compact cards with four visible before a show-more control. Document cards show role, size, and relative modified time, preferring file-level `updated_at` when the registry provides it and otherwise falling back to the artifact version timestamp. Each listed document opens a document inspector modal, and Markdown documents render through the same `react-markdown` + `remark-gfm` stack used by assistant-ui markdown guidance. If real registry metadata exists but the local blob body is missing or unavailable, the inspector shows an explicit storage-unavailable message instead of sample content or a misleading empty-Markdown placeholder. Fenced Mermaid blocks render with render/source controls; other fenced code blocks render as code. The inspector is the phase-one place for document-centric inspection: View mode has Markdown and Code tabs plus a copy action for the current document body, and Diff mode compares the selected version against the latest version with a line-level diff. Feature attachments are shown read-only in the artifact modal when the artifact is linked to a Feature: `gate`/`both` rows are consumed by quality gates, and `coding_agent`/`both` rows are rendered into Context Packs for IDE agents. Attachments are never created implicitly from IDE-agent chat uploads; they are explicit governed pins with an audience. Expected gate rows show the shared gate-catalog sentence (what the gate verifies and what it reads) under the gate name, and the registry's placeholder preview note ("preview — not persisted") is replaced with "Expected for this artifact's profile. Not run yet."; real registry notes pass through unchanged. The Readiness history section opens with "Each run records what the checker read and why it decided.", and each readiness run with parseable `evidence_json` gets the same "Why" disclosure as work-item gate runs (evaluator origin, confidence when judged, quoted or per-topic/per-criterion evidence; nothing on parse failure — never raw JSON). Persisted readiness runs, saved revisions, audit events, expected gate previews, and policy explanation are read-only quality/provenance evidence; browser rerun/refresh/gate-dispatch/revision-apply/exception-acceptance controls remain absent because `POST /artifacts/{id}/readiness-runs/refresh`, `POST /api/v1/artifacts/{id}/gate-tasks`, `POST /api/v1/artifacts/{id}/accepted-exceptions`, and artifact-edit save/verdict paths are execution paths for CLI/IDE-agent/backend owners. If a file changes source-of-truth content, it belongs in a reviewed artifact proposal rather than the attachment list. Attachment upload/delete/audience mutation, readiness reruns, gate task dispatch, conflict/proposal review, accepted-exception recording, and feedback triage stay deferred until those workflow owners are explicit. The artifact list is a compact full-width table with artifact, version, impact, type, and updated columns.

Document inspector Copy/Diff actions stay disabled when a live registry document body is unavailable, so the browser does not copy an empty placeholder or compare fake content.

Document inspector version-history failures keep Diff disabled and show an explicit live-mode error instead of comparing against fallback version data.

Artifact document-list failures show an explicit live-mode error instead of empty-document copy or sample document cards.

Artifact gate-preview failures show an explicit live-mode error and do not render fallback gates from artifact list metadata as a substitute for `GET /api/v1/artifacts/{id}/gate-preview`.

Artifact detail evidence-section failures for attachments, feedback, readiness history, saved revisions, and audit events show explicit live-mode errors instead of empty-record copy or sample evidence.

Artifact linked-feature and policy readback failures show explicit live-mode errors instead of hiding feature context or showing generic policy absence.

The Reviews surface derives delivery-review counts and queue rows from the same workboard items rather than static cards. Delivered items (server-derived `delivered` phase) are excluded from the review queue and its count entirely — they already passed review; passed-but-not-delivered items keep their existing Passed labeling. The review-count heading pluralizes correctly (`1 item needs review` / `N items need review`). It filters by all reviews, needs changes, ready for review, and gate failed, and supports forgiving search across work item, reason, route, verdict, Context Pack identifiers, artifact-update proposal metadata, and governance feedback signals. The review queue table has no Owner column. The page stays a dense triage index rather than a card-plus-side-rail dashboard. The delivery-review table matches the compact Work and Artifact table language: status, route, and Context Pack evidence appear in the item cell, updated time comes before actions, and action controls are icon-only with accessible labels and tooltips. Clicking the row, item key, or primary review action navigates straight to the work item's Verification tab (`/work/:key?tab=verification`), where the delivery verdict renders first. `src/data/reviews.ts` also reads `GET /artifact-edit/proposals`, `GET /artifact-edit/sessions/{id}/diff`, and `GET /governance/feedback-events?status=received&limit=20` so Reviews can show pending artifact-update proposals, inspect their unified diff, and surface received CLI/IDE-agent/integration feedback signals. Proposal rows require a real registry proposal id before they can appear or fetch a diff; the browser must not synthesize proposal ids from incomplete rows. Feedback rows require a real registry feedback id before rendering; Work and Artifact links appear only when the registry provides those identifiers, and the UI must not call `POST /governance/feedback-events/{id}/status` until feedback triage ownership and status semantics are explicit. The proposal diff modal renders file/hunk markers as patch structure and changed Markdown lines as readable Markdown with add/remove coloring and wrapping. Proposal rows carry the human decision: approve saves the session as a draft revision and reject discards it after confirmation, both attributed to the local user; feedback triage and conflict resolution remain deferred. Governance-agent prompt buttons remain advisory and use the robot icon. Empty filtered views link back to Work. Reviews, proposal diffs, and feedback signals render only real Doc Registry data. Missing registry configuration, request failures, and empty responses show setup/live-mode placeholders and do not fall back to bundled sample rows or sample diffs.

## Settings

Settings opens from the sidebar footer as a non-blocking modal configuration surface with a fixed viewport-relative height, a vertical section menu for General, Models, Governance, Plugins, and Integrations (Experimental), and a fixed footer. Settings has no Workspace or Knowledge section: workspace/user identity actions live in the sidebar account menu, and the browser exposes no knowledge UI while the Doc Registry knowledge endpoints remain backend contracts. On mobile, Settings behaves as small pages: the section list is the first page, selecting a section opens that section, and Back returns to the section list. Desktop keeps the two-column section list plus content layout. Settings is not a standalone route; direct `/settings` visits redirect to `/work`. Query `?settings=<section>` opens the matching Settings section when it exists, then the shell clears the query. Unknown section queries (including the `workspace` and `knowledge` ids) resolve to nothing and Settings keeps its current section, General by default. Close is always in the footer. Save actions, when backed by a real flow, also live in the footer rather than inside the active settings panel. Nested Settings dialogs, such as Skill detail/edit and Add integration, keep footer actions fully visible with a scrollable body on mobile and desktop. The right pane is an unframed scrollable content area, not a nested card, and long registry text wraps inside the pane instead of horizontally clipping the modal. The global governance agent launcher remains available while Settings is open. Settings and the governance composer must not show static sample work, artifact, integration, tool, resource, or mention catalogs. Settings must not show save, refresh, add, or provider-selection controls unless they call a real local/backend configuration flow.

Plugins is a static help card, not a status surface. Because the browser cannot inspect local IDE plugin files, the section shows one explanatory sentence — IDE plugins are installed and updated via the CLI — plus copyable `specgate plugins install`, `specgate plugins doctor`, and `specgate update` commands. It renders no per-agent rows, versions, or pseudo-live status badges; plugin refresh/status detection remains a CLI or backend-owned action. Installed IDE plugin Skills are treated as bundled plugin contents and are not listed as standalone mutable browser rows. MCP tools/resources from `/mcp/info` stay out of browser Settings because they are CLI/IDE-agent execution context, not user configuration. The browser must not install plugins, fetch MCP access tokens, rotate tokens, show tool/resource catalogs, execute MCP tools, or manage team rubric Skills from Plugins.

General settings are a narrow editable surface for user-facing governance defaults backed by Doc Registry `GET /settings` and `PUT /settings`. The tab edits only `governance.auto_archive_on_delivery_pass`, `governance.feature_freshness_sla_days`, and `governance.artifact_stale_days`; Save sends only those keys. `governance.auto_archive_on_delivery_pass` defaults to `"false"` and lets Doc Registry archive a work item only after a future persisted `delivery_review` gate run passes, with archive metadata written by the backend. Auto-archive hides completed work from the default Work list; it does not delete work, artifacts, evidence, or chat threads. `governancefiles.ttl_days`, confidence thresholds, publish retry/timeout knobs, MCP token/config, speech-to-text settings, and policy rollback flags stay operator-owned and out of the browser surface entirely — General renders no read-only explainer cards for them — because they affect retention, governance verdicts, transport behavior, credentials, or operator rollback. Manual archive belongs in Work/CLI flows with actor, reason, and audit context; Settings must not expose generic archive/delete controls.

Model settings reserve separate surfaces for the server-side model (which powers governance operations — readiness gates, route classification, summaries, and delivery review; the governance chat agent's model is configured separately via environment) and the embedding model. The Models tab is backed by Doc Registry `GET /settings` and `PUT /settings` when `VITE_DOC_REGISTRY_URL` is configured. It edits `governance.model_provider`, `governance.model`, `governance.default_thinking_level`, `embedding.model_provider`, `embedding.model`, and the selected provider API keys while preserving masked `***` secrets. Provider choices use the UI's built-in provider brand icons. OpenRouter uses its public models API (`/api/v1/models?sort=most-popular&output_modalities=text`) to populate the governance searchable text-model picker, filters out image/audio-output models client-side, falls back to seeded vendor/model slugs when the catalog is unavailable, and allows custom `vendor/model` entry. Saved custom slugs remain visible as the selected option even when they are absent from the current catalog response. The footer Save action is enabled only after the loaded model settings change. The embedding section uses the same pattern with `output_modalities=embeddings`, keeps embeddings disabled by default, and only shows model/key controls after an embedding provider is selected.

Governance settings lead with team rubric Skill management — the panel's only durable actions — followed by a compact read-only Policy catalog section. A single note at the section top states that the catalog is read-only; catalog rows do not repeat per-row read-only badges. When `VITE_DOC_REGISTRY_URL` is configured, the section reads `GET /api/v1/skills` for Skills and, for the catalog, `GET /governance-profiles` and `GET /api/v1/policies/levels`, showing profile names, descriptions, mapped work types, required checks, policy levels, LLM gate settings, delivery-review requirements, enabled-gate lists rendered with the shared gate-catalog name, the raw gate key as a mono suffix, and the catalog's what-it-checks sentence (unknown keys fall back to the readable key alone), and team rubric Skills. Team rubric Skills are governance rubrics that may appear in gates and Context Packs; they are not proof that a local IDE plugin is installed and do not show a Registry badge. Live governance profiles require `full_key` or `namespace` plus `key`, and live policy levels require `governance_level`; incomplete rows are ignored instead of becoming fake `profile-*` or `level-*` catalog entries. Live Skill rows require a real registry `id` before rendering or enabling inspect/edit/delete actions; the browser must not synthesize Skill ids from names or row indexes. Selecting a Skill loads `GET /api/v1/skills/{id}` and keeps edit controls inline in the detail dialog: prompt and description become text areas, and save uses the existing `/skills/{id}` update route. Users can add and delete Skills through the existing `/skills` create/delete routes after explicit user action. Policy/profile catalog failures do not hide team rubric Skill management when the Skills endpoint remains available. It is mostly inspection: profile activation, policy import/export, dry-run, exception workflows, and backend policy edits stay outside the browser until an explicit governance-admin owner exists.

Integrations (Experimental) contains tracker and repository adapter status with provider icons and plain text about the backend owner of each connection. When `VITE_DOC_REGISTRY_URL` is configured, Settings can add GitHub, GitLab, or Linear integrations through Doc Registry `/integrations`, store a write-only provider API token through `/integrations/{id}/api-token`, or start the provider OAuth redirect through `/integrations/oauth/begin`. OAuth redirects return to the current app path with `?settings=integrations` so the Integrations section reopens after authorization, but the browser only redirects after Doc Registry returns a non-empty `authorize_url`; missing redirect responses show an error and keep the dialog open. Hosted GitHub/GitLab and OAuth flows do not ask for Base URL; Base URL is shown only for API-token setup after the user marks GitHub/GitLab as self-hosted. Live integration rows require a real registry `id` before rendering or fetching resources/webhook events; missing registry configuration, empty responses, and request failures show placeholders instead of static GitHub/Linear sample rows. Per-integration details are lazy: opening the section fetches only `GET /integrations`, and each integration's linked resources and recent webhook events load the first time that integration's details are expanded via its Show details toggle (or immediately after the UI creates that integration or links a resource through it), never eagerly for every row. Existing linked resources are shown read-only from `GET /integrations/{id}/resources`, including resource labels, default refs, stored webhook-secret presence, and provisioning metadata such as `config_json.webhook_status`. A connected integration can link one provider resource through the backend-owned picker flow: GitHub/GitLab use `GET /integrations/{id}/repos?limit=50` plus `POST /integrations/{id}/resources` with `resource_type=project`, and Linear uses `GET /integrations/{id}/linear/teams` plus `POST /integrations/{id}/resources` with `resource_type=team`. Provider candidates must include a real non-empty `external_key`; incomplete candidate rows are hidden and cannot enable Link. Linear project selection remains deferred because outbound tracker handoff currently resolves issue creation at team scope. The browser submits only the selected provider identifier/default ref/display name; webhook provisioning, when possible, is owned by Doc Registry and reflected in the returned resource row. Recent webhook deliveries are shown read-only from `GET /integrations/{id}/webhook-events?limit=3` so humans can see whether provider signals are reaching the registry. This is configuration/provisioning and inbox visibility only, not a full integration-admin surface. Webhook reprovisioning, disconnect, webhook-secret reveal/rotation, resource delete, project-scoped Linear handoff, and destructive integration delete remain deferred to fuller backend-owned management flows.
